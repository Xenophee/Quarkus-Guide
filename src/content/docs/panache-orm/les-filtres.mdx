Dans cet article, nous allons explorer des fonctionnalités avancées de **Panache ORM** dans **Quarkus**, en mettant l'accent sur l'utilisation des filtres pour affiner les opérations CRUD (Create, Read, Update, Delete). Les filtres permettent de définir des critères spécifiques pour les opérations, offrant ainsi une flexibilité accrue dans la gestion des entités.

<Aside type="note">
À partir de maintenant, les exemples de code se baseront sur le pattern **Repository** uniquement afin de garder en tête les bonnes pratiques. L'équivalent en **Active Record** est très similaire.
</Aside>


## Création d'une entité

Pour commencer, nous devons définir une entité qui représentera une table dans notre base de données. Voici un exemple d’entité `Alien` pour illustrer les opérations.

```java title="Alien.java"
@Entity
@Table(name = "aliens")
public class Alien {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String codename;

    private int healthPoints;

    private int attackPower;

    private int stealthLevel;

    // Getters et Setters

}
```

Puis la classe **repository** associée :

```java title="AlienRepository.java"
@ApplicationScoped
public class AlienRepository implements PanacheRepository<Alien> {
    // Méthodes CRUD personnalisées sont ajoutées ici
}
```

<HR />

## Utilisation des filtres avec Panache ORM

Avant toute chose, il faut savoir qu'il existe deux façons d'indiquer des filtres dans les méthodes **Panache** :

1. En utilisant des paramètres positionnels (exemple : `?1`, `?2`, etc.)
2. En utilisant des paramètres nommés (exemple : `:paramName`)

Nous allons voir des exemples pour les deux méthodes.

<Aside type="tip">
Le premier est plus simple à utiliser, mais requiert de faire attention à l'ordre des paramètres. Le second est plus verbeux, mais moins sujet aux erreurs lorsque plusieurs paramètres sont utilisés.

Les bonnes pratiques recommandent d'utiliser les **paramètres nommés** pour des requêtes complexes.
</Aside>

<Aside type="caution">
    L'écriture des paramètres dans les **ORM** se basent sur les **entités** et leurs **attributs**, et non sur les noms des colonnes en base de données. <Underline>Assurez-vous d'utiliser les noms des attributs tels qu'ils sont définis dans la classe de l'entité.</Underline>

    Par exemple, si l'attribut dans l'entité est `healthPoints`, utilisez ce nom dans vos filtres, même si la colonne en base de données s'appelle `health_points`.
</Aside>


### <i class="ri-search-line"></i> Lecture (Read)

Pour lire des entités en utilisant des filtres, vous pouvez utiliser la méthode `find` de Panache avec une clause `where` sans avoir besoin de l'ajouter explicitement.

**Pour récupérer un alien en fonction de son nom de code :**

<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public Alien getAlienByCodename(String codename) {
        return find("codename = ?1", codename).firstResult();
    }
    ```

    ou si l'on veut encapsuler un `NULL` potentiel :

    ```java {2} title="AlienRepository.java"
    public Optional<Alien> getAlienByCodename(String codename) {
        return find("codename = ?1", codename).firstResultOptional();
    }
    ```

    Le paramètre `?1` est remplacé par la valeur fournie dans l'appel de la méthode.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public Alien getAlienByCodename(String codename) {
        return find("codename = :name", Parameters.with("name", codename)).firstResult();
    }
    ```

    ou si l'on veut encapsuler un `NULL` potentiel :

    ```java {2-3} title="AlienRepository.java"
    public Optional<Alien> getAlienByCodename(String codename) {
        return find("codename = :name", Parameters.with("name", codename))
            .firstResultOptional();
    }
    ```

    Le paramètre `:name` est remplacé par la valeur fournie dans l'appel de la méthode.

    </TabItem>
</Tabs>

<Aside type="tip">
Pour éviter les erreurs de type `NullPointerException` lorsque vous manipulez le résultat, il est recommandé d'utiliser la méthode `firstResultOptional()` qui renvoie un `Optional<Alien>`. Cela permet de gérer plus facilement et plus proprement les cas où aucun résultat n'est trouvé.
</Aside>


<HR />

**Pour récupérer tous les aliens dont la puissance d'attaque est supérieure à un nombre donné :**

<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public List<Alien> getPraetorianAlien(int minAttackPower) {
        return find("attackPower > ?1", minAttackPower).list();
    }
    ```

    Pour plus de concision, on peut aussi utiliser la méthode `list()` directement :

    ```java {2} title="AlienRepository.java"
    public List<Alien> getPraetorianAlien(int minAttackPower) {
        return list("attackPower > ?1", minAttackPower);
    }
    ```

    Le paramètre `?1` est remplacé par la valeur fournie dans l'appel de la méthode.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public List<Alien> getPraetorianAlien(int minAttackPower) {
        return find("attackPower > :power", Parameters.with("power", minAttackPower)).list();
    }
    ```

    Pour plus de concision, on peut aussi utiliser la méthode `list()` directement :

    ```java {2} title="AlienRepository.java"
    public List<Alien> getPraetorianAlien(int minAttackPower) {
        return list("attackPower > :power", Parameters.with("power", minAttackPower));
    }
    ```

    Le paramètre `:power` est remplacé par la valeur fournie dans l'appel de la méthode.

    </TabItem>
</Tabs>

<HR />

Si vous avez des critères multiples, vous pouvez les ajouter sans problème en utilisant les opérateurs logiques appropriés (`AND`, `OR`, etc.) comme si vous écriviez une **requête SQL classique**.

**Pour compter le nombre d'aliens dont les points de vie sont entre deux valeurs données :**

<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public long getHealthyAlien(int minHealth, int maxHealth) {
        return count("healthPoints between ?1 and ?2", minHealth, maxHealth);
    }
    ```

    Les paramètres `?1` et `?2` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2-3} title="AlienRepository.java"
    public long getHealthyAlien(int minHealth, int maxHealth) {
        return count("healthPoints between :min and :max",
            Parameters.with("min", minHealth).and("max", maxHealth));
    }
    ```

    Les paramètres `:min` et `:max` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>
</Tabs>




### <i class="ri-edit-2-line"></i> Mise à jour (Update)

Pour mettre à jour des entités en utilisant des filtres, vous pouvez utiliser la méthode `update` de Panache avec une clause `where`.

**Pour mettre à jour les points de vie de tous les aliens dont le niveau de furtivité est supérieur à une valeur donnée :**


<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public int updateSneakyAlien(int healthPoints, int stealthLevel) {
        return update("healthPoints = ?1 where stealthLevel > ?2", healthPoints, stealthLevel);
    }
    ```

    Les paramètres `?1` et `?2` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2-3} title="AlienRepository.java"
    public int updateSneakyAlien(int healthPoints, int stealthLevel) {
        return update("healthPoints = :health where stealthLevel > :stealth",
            Parameters.with("health", healthPoints).and("stealth", stealthLevel));
    }
    ```

    Les paramètres `:health` et `:stealth` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>
</Tabs>

<HR />

Si vous avez des critères multiples, vous pouvez les ajouter sans problème en utilisant les opérateurs logiques appropriés (`AND`, `OR`, etc.) comme si vous écriviez une **requête SQL classique**.

**Pour mettre à jour la puissance d'attaque de tous les aliens dont les points de vie sont inférieurs à une valeur donnée et le niveau de furtivité est supérieur à une autre valeur indiquée :**

<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public int updateAttackPowerForSpecificAlien(int attackPower, int healthPoints, int stealthLevel) {
        return update("attackPower = ?1 where healthPoints < ?2 and stealthLevel > ?3", attackPower, healthPoints, stealthLevel);
    }
    ```

    Les paramètres `?1`, `?2` et `?3` sont remplacés par les valeurs fournies dans l'appel de la méthode. Avec trois paramètres, l'utilisation des paramètres nommés est recommandée pour éviter toute confusion.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2-5} title="AlienRepository.java"
    public int updateAttackPowerForSpecificAlien(int attackPower, int healthPoints, int stealthLevel) {
        return update("attackPower = :power where healthPoints < :health and stealthLevel > :stealth",
            Parameters.with("power", attackPower)
                .and("health", healthPoints)
                .and("stealth", stealthLevel));
    }
    ```
    Les paramètres `:power`, `:health` et `:stealth` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>
</Tabs>

<HR />

<Aside type="tip">
Elle est indispensable pour les mises à jour multiples, mais aussi pour une entité unique en évitant de charger l'entité au préalable. Il suffit de s'assurer que le filtre utilisé dans la clause `where` identifie de manière unique l'entité à mettre à jour (par exemple, en utilisant son ID). Gain de performance garanti !

Notez également que la méthode `update` retourne un `int` représentant le nombre d'enregistrements affectés par la mise à jour, ce qui peut être utile pour vérifier si l'opération a réussi.
</Aside>

<Aside type="caution">
    La méthode `update` avec des filtres est puissante, mais elle doit être utilisée avec précaution pour éviter des mises à jour accidentelles de plusieurs enregistrements. Assurez-vous toujours de bien définir vos critères dans la clause `where`.
</Aside>


### <i class="ri-delete-bin-2-line"></i> Suppression (Delete)

Pour supprimer des entités en utilisant des filtres, vous pouvez utiliser la méthode `delete` de **Panache** avec une clause `where` sans avoir besoin de l'ajouter explicitement.

**Pour supprimer tous les aliens dont le niveau de furtivité est égal à une valeur donnée :**

<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public long deleteAlienBySneakiness(int stealth) {
        return delete("stealthLevel = ?1", stealth);
    }
    ```

    Le paramètre `?1` est remplacé par la valeur fournie dans l'appel de la méthode.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public long deleteAlienBySneakiness(int stealth) {
        return delete("stealthLevel = :level", Parameters.with("level", stealth));
    }
    ```

    Le paramètre `:level` est remplacé par la valeur fournie dans l'appel de la méthode.

    </TabItem>
</Tabs>


<HR/>


Si vous avez des critères multiples, vous pouvez les ajouter sans problème en utilisant les opérateurs logiques appropriés (`AND`, `OR`, etc.) comme si vous écriviez une **requête SQL classique**.

**Pour supprimer tous les aliens dont la puissance d'attaque est inférieure à une valeur donnée et les points de vie différents d'une autre valeur indiquée :**

<Tabs>
    <TabItem label="Paramètres positionnels" icon="right-caret">

    ```java {2} title="AlienRepository.java"
    public long deleteAlienWithBelowAttackAndNotHealthPoints(int attackPower, int health) {
        return delete("attackPower < ?1 and healthPoints <> ?2", attackPower, health);
    }
    ```
    Les paramètres `?1` et `?2` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>

    <TabItem label="Paramètres nommés" icon="right-caret">

    ```java {2-3} title="AlienRepository.java"
    public long deleteAlienWithBelowAttackAndNotHealthPoints(int attackPower, int health) {
        return delete("attackPower < :power and healthPoints <> :health",
            Parameters.with("power", attackPower).and("health", health));
    }
    ```

    Les paramètres `:power` et `:health` sont remplacés par les valeurs fournies dans l'appel de la méthode.

    </TabItem>
</Tabs>

<HR />

<Aside type="tip">
    La méthode `delete` retourne un `long` représentant le nombre d'enregistrements supprimés, ce qui peut être utile pour vérifier si l'opération a réussi.
</Aside>

<Aside type="caution">
La méthode `delete` avec des filtres est très puissante, mais elle doit être utilisée avec précaution pour éviter de supprimer accidentellement des données importantes. Assurez-vous toujours de bien définir vos critères dans la clause `where`, car elle peut supprimer plusieurs enregistrements à la fois.
</Aside>
