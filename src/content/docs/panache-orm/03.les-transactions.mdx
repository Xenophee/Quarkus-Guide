## Rappel sur les transactions

Une **transaction** est une **unité de travail cohérente** regroupant plusieurs opérations qui doivent être exécutées comme un tout.

Si toutes les opérations réussissent → la transaction est **validée** (*commit*).

Si l’une d’elles échoue → l’ensemble est **annulé** (*rollback*), garantissant l’intégrité des données.

<Aside type="note">
    Les transactions reposent traditionnellement sur le modèle **ACID** :

    - **Atomicité** – La transaction forme un bloc indivisible : tout réussit ou tout échoue.
    - **Cohérence** – La base passe d’un état valide à un autre état valide.
    - **Isolation** – Une transaction en cours ne montre pas ses modifications aux autres tant qu’elle n’est pas validée.
    - **Durabilité** – Une fois validées, les modifications persistent même en cas de panne.
</Aside>


En pratique, une transaction suit souvent ce schéma :

- **début de transaction**
- opérations de lecture/écriture
- **commit** ou **rollback**

Dans **Quarkus**, cette logique est prise en charge par **Jakarta Transactions** (*JTA*), et **Panache** vient simplifier son utilisation au quotidien.

## Gestion des transactions avec Panache

**Quarkus** ne démarre pas automatiquement une transaction pour les opérations ORM classiques : c’est au développeur de déclarer explicitement quand un bloc de code doit être transactionnel.

**Panache** propose deux façons principales de gérer les transactions :

1. Annoter une méthode avec `@Transactional`.
2. Contrôler manuellement une transaction via `Panache.withTransaction()`.
3. Utiliser l'`EntityManager` d'**Hibernate** pour une gestion fine des transactions.


### Avec l'annotation `@Transactional`

L’annotation `@Transactional` permet de marquer une méthode comme transactionnelle. Lorsque cette méthode est appelée, Quarkus démarre automatiquement une transaction avant l’exécution et la valide ou l’annule à la fin. Par exemple :

```java
@Transactional
public void createAstronaut(String name) {
    Astronaut astro = new Astronaut();
    astro.setName(name);
    astro.persist();
}
```
Ici, si `createAstronaut` s’exécute sans erreurs, la transaction est validée. En cas d’exception, elle est annulée.

<hr />

Il existe plusieurs options pour configurer le comportement transactionnel via `@Transactional`, comme le niveau d’isolation, le délai d’attente, etc.
**Quarkus** s'appuyant sur l'implémentation de **Jakarta Transactions** pour gérer ces options, il n'y a pas autant d'options personnalisables que dans certains autres frameworks. L'annotation `@Transactional` supporte les options suivantes :

1. **Transactional.TxType** : permet de spécifier le type de transaction à utiliser. Les constantes disponibles sont :
    - `REQUIRED` : utilise la transaction existante ou en crée une nouvelle si aucune n’existe.
    - `REQUIRES_NEW` : suspend la transaction existante et en crée une nouvelle.
    - `MANDATORY` : exige qu’une transaction existe déjà, sinon une exception est levée.
    - `SUPPORTS` : exécute dans le contexte de la transaction existante si elle existe, sinon sans transaction.
    - `NOT_SUPPORTED` : suspend toute transaction existante et exécute sans transaction.
    - `NEVER` : exécute sans transaction et lève une exception si une transaction existe.

<Aside type="tip">
    Par défaut, `@Transactional` utilise `TxType.REQUIRED`, ce qui convient à la plupart des cas courants.
</Aside>

2. **rollbackOn** : permet de spécifier les types d’**exceptions** qui doivent déclencher un rollback. Par exemple, `@Transactional(rollbackOn = {CustomException.class})` indique que si `CustomException` est levée, la transaction doit être annulée.

3. **dontRollbackOn** : permet de spécifier les types d’**exceptions** qui ne doivent pas déclencher un rollback. Par exemple, `@Transactional(dontRollbackOn = {CustomException.class})` indique que si `CustomException` est levée, la transaction ne doit pas être annulée.

<HR />

### Avec `Panache.withTransaction()`

Pour des cas plus complexes, `Panache.withTransaction()` permet de définir un bloc de code transactionnel de manière programmatique. Par exemple :

```java
Panache.withTransaction(() -> {
    Astronaut astro = new Astronaut();
    astro.setName("Dallas");
    astro.persist();
    // autres opérations
});
```
Ici, toutes les opérations à l’intérieur du bloc seront exécutées dans une **transaction**. Si une **exception** est levée, la **transaction** sera annulée.

<HR />

### Avec l'`EntityManager`

Même si **Panache** simplifie grandement la gestion des **transactions**, il est toujours possible d’utiliser directement l’`EntityManager` pour un contrôle plus fin. Par exemple :

```java

public void customTransaction() {

    EntityManager em = Panache.getEntityManager();
    EntityTransaction transaction = em.getTransaction();

    try {
        transaction.begin();
        // opérations de persistance
        transaction.commit();
    } catch (Exception e) {
        if (transaction.isActive()) transaction.rollback();
        throw e;
    }

}
```

<HR />

### Bonnes pratiques

<Aside type="tip">
    Dans la plupart des cas, préférez l’utilisation de l’annotation `@Transactional` pour sa simplicité et sa clarté. Utilisez `Panache.withTransaction()` uniquement lorsque vous avez des besoins spécifiques qui ne peuvent pas être couverts par l’annotation.
</Aside>

<Aside type="caution">
    Attention à ne pas imbriquer des **transactions** de manière incorrecte, ce qui pourrait entraîner des comportements inattendus.

    Evitez également d’effectuer des opérations non liées à la base de données à l’intérieur d’une **transaction** pour ne pas prolonger inutilement sa durée et améliorer les performances.

    Les transactions <Underline>doivent rester focalisées sur les opérations de persistance</Underline> ; pour les simples méthodes de lecture, il n’est généralement pas nécessaire d’utiliser une transaction explicite.

    Pensez à bien gérer les **exceptions** pour éviter des **rollbacks** inattendus.
</Aside>
