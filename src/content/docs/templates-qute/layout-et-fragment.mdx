Dans l'article précédent, nous avons vu comment créer des templates simples avec **Qute**. Cependant, à mesure que votre application grandit, il devient essentiel de structurer vos templates de manière efficace pour éviter la duplication de code et faciliter la maintenance. C'est là que les **sections** et les **fragments** entrent en jeu.


## Layout

Dans une application, il arrive régulièrement que certaines parties de l'interface utilisateur soient communes à plusieurs pages, comme les en-têtes, les pieds de page ou les barres de navigation. Plutôt que de dupliquer ce code dans chaque template, vous pouvez définir ces parties communes comme des **templates** réutilisables.

Basiquement, il s'agit de créer des blocs de contenu dans un **template** qui peuvent être inclus dans d'autres **templates**. Cela permet de centraliser la gestion de ces éléments communs et de les modifier facilement à un seul endroit.

Pour commencer, vous pouvez créer un template de base, par exemple `base.html`, qui contient la structure générale de votre page, y compris les sections communes. Voici un exemple de ce à quoi pourrait ressembler ce **template** de base :

```html title="base.html"
<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>Weyland-Yutani</title>
</head>
<body>
    <header>
        <h1>Bienvenue chez Weyland-Yutani</h1>
    </header>
    <main>
        <p>Dans l'espace, personne ne vous entend crier...</p>
    </main>
    <footer>
        <p>Tous droits réservés © Weyland-Yutani Corporation</p>
    </footer>
</body>
</html>
```

Dans ce **template**, le **header** et le **footer** sont des éléments qui peuvent être amenés à être réutilisés dans d'autres pages. L'objectif serait donc de les extraire dans des sections séparées pour pouvoir les inclure facilement dans d'autres **templates**.


### La commande `#include`

Pour ce faire, vous pouvez créer des fichiers de **template** dédiés pour ces sections, par exemple `header.html` et `footer.html`, et y placer le code correspondant. Ensuite, dans votre **template** de base, vous pouvez inclure ces sections à l'aide de la syntaxe d'inclusion de **Qute**, `#include`. Par exemple, votre `base.html` pourrait ressembler à ceci après l'inclusion des sections :

```html {8,12} title="base.html"
<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>Weyland-Yutani</title>
</head>
<body>
    {#include header /}
    <main>
        <p>Dans l'espace, personne ne vous entend crier...</p>
    </main>
    {#include footer /}
</body>
</html>
```


### La commande `#insert`

Très souvent, la partie centrale de votre page (le `main`) contiendra du contenu spécifique à chaque page. Vous pouvez y mettre un contenu par défaut entre les balises ou laisser un espace vide que les autres **templates** puissent y injecter leur propre contenu. Par exemple, vous pouvez définir une section dans votre **template** de base pour le contenu principal avec une syntaxe spéciale `#insert`, comme ceci :

```html {10,11} title="base.html"
<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>Weyland-Yutani</title>
</head>
<body>
    {#include header /}
    <main>
        <h1>{#insert title}Bienvenue sur le Nostromo{/}</h1>
        {#insert } {/}
    </main>
    {#include footer /}
</body>
</html>
```

Dans cet exemple, nous avons défini deux sections d'insertion : une pour le titre de la page et une autre pour le contenu principal. Les autres **templates** qui incluent ce `base.html` comme squelette pourront alors injecter leur propre contenu dans ces sections d'insertion, ce qui permet de personnaliser chaque page tout en réutilisant la structure commune définie dans le template de base.

Le `#insert` sans plus de précision correspond à la section d'insertion principale, tandis que le `#insert title` correspond à une section d'insertion spécifique pour le titre de la page. `title` est un nom arbitraire que nous avons choisi pour identifier cette section d'insertion, mais vous pouvez en créer autant que nécessaire avec des noms différents pour organiser votre contenu de manière plus structurée.

Pour le template de la page d'accueil qui réutiliserait `base.html` par exemple, ça pourrait donner ceci :

```html {3,5} title="home.html"
{#include base}

    {#title}J'ai décidé de redéfinir le titre par défaut{/}

    <p>Je suis un paragraphe perdu dans l'espace</p>
{/}
```

Dans ce **template**, nous incluons `base.html` et redéfinissons le contenu de la section `title` pour personnaliser le titre de la page d'accueil. Nous injectons également du contenu dans la section d'insertion principale pour afficher un paragraphe spécifique à cette page.

Vous pouvez également redéfinir le titre en passant par la mécanique des **expressions**, en utilisant des **variables** pour rendre le **template** plus dynamique. Par exemple, dans votre `base.html`, vous pouvez remplacer la section d'insertion du titre par une expression qui utilise une **variable**, comme ceci :

```html {10} title="base.html"
<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>Weyland-Yutani</title>
</head>
<body>
    {#include header /}
    <main>
        <h1>{title}</h1>
        {#insert } {/}
    </main>
    {#include footer /}
</body>
</html>
```

Ensuite, dans votre `home.html`, vous pouvez passer une valeur pour la **variable** `title` lors de l'inclusion du template de base, comme ceci :

```html "title="Je remplace le titre"" title="home.html"
{#include base title="Je remplace le titre" /}
    <p>Je suis un paragraphe perdu dans l'espace</p>
{/}
```
Le paragraphe, quant à lui, est injecté dans la section d'insertion principale du `base.html`, ce qui permet de personnaliser le contenu de la page d'accueil tout en réutilisant la structure commune définie dans le template de base.

## Fragments

Les **fragments** sont des morceaux de code contenu dans des **templates** qui peuvent être rendu séparément du reste du template. Cette mécanique est particulièrement utile pour les composants dynamiques basés sur de l'**Ajax**. Par exemple, vous pouvez avoir un template `astronaut.html` qui contient un fragment pour afficher les informations d'un astronaute.

Ce fragment peut être rendu indépendamment du reste du **template**, ce qui permet de mettre à jour dynamiquement la liste des astronautes sans avoir à recharger toute la page.

### Déclaration d'un fragment

Pour déclarer un **fragment** dans un **template**, vous pouvez utiliser la syntaxe `#fragment`, comme ceci :

```html {3,5} title="astronaut.html"
<ul id="astroList">
    {#for astro in astronauts}
        {#fragment id=astroItem}
            <li>{astro.name}</li>
        {/fragment}
    {/for}
</ul>
```

Le rendu du **template** `astronaut.html` affichera la liste des astronautes, mais le **fragment** `astroItem` peut être rendu séparément pour mettre à jour la liste de manière dynamique. Par exemple, vous pouvez créer un **endpoint REST** qui rend ce **fragment** pour ajouter un nouvel astronaute à la liste sans recharger la page entière.


### Rendu d'un fragment

    ```java title="AstronautResource.java"
    public class AstronautResource {

        @CheckedTemplate
        public static class Templates {
            public static native TemplateInstance astronaut(List<Astronaut> astronauts);
            public static native TemplateInstance astronaut$astroItem(Astronaut astro);
        }

        @GET
        public TemplateInstance getAstronauts() {
            List<Astronaut> astronauts = // Récupérer la liste des astronautes (logique de récupération non montrée ici)
            return Templates.astronaut(astronauts);
        }


        @POST
        @Path("/add")
        public TemplateInstance addAstronaut(@FormParam("name") String name) {
            Astronaut astro = new Astronaut(name);
            // Ajouter l'astronaute à la liste (logique de stockage non montrée ici)
            return Templates.astronaut$astroItem(astro);
        }
    }
    ```

    Pour rendre un **fragment** de manière **type-safety**, vous pouvez créer une méthode spécifique dans votre classe de ressource annotée avec `@CheckedTemplate` qui correspond au **fragment** que vous souhaitez rendre.

    Par exemple, dans l'exemple ci-dessus, nous avons créé une méthode `home$astroItem` qui correspond au **fragment** `astroItem` défini dans le **template** `home.html` en utilisant le symbole `$` pour indiquer qu'il s'agit d'un **fragment**. Cette méthode prend un objet `Astronaut` en paramètre et retourne une instance de `TemplateInstance` pour ce **fragment**. Ensuite, dans la méthode `addAstronaut`, nous appelons cette méthode pour rendre le **fragment** avec les données de l'astronaute nouvellement ajouté.

    </TabItem>

    ```java title="AstronautResource.java"
    @Path("/")
    public class AstronautResource {

        private final Template astronaut;

        @Inject
        public AstronautResource(Template astronaut) {
            this.astronaut = astronaut;
        }

        @GET
        public TemplateInstance getAstronauts() {
            List<Astronaut> astronauts = // Récupérer la liste des astronautes (logique de récupération non montrée ici)
            return astronaut.data("astronauts", astronauts);
        }

        @POST
        @Path("/add")
        public TemplateInstance addAstronaut(@FormParam("name") String name) {
            Astronaut astro = new Astronaut(name);
            // Ajouter l'astronaute à la liste (logique de stockage non montrée ici)
            return astronaut.getFragment("astroItem").data("astro", astro);
        }
    }
    ```

    Pour rendre un **fragment** de manière non **type-safety**, vous pouvez utiliser la méthode `getFragment` de l'instance de `Template` pour récupérer le **fragment** que vous souhaitez rendre, puis passer les données nécessaires à ce **fragment** en utilisant la méthode `data`.

    Dans l'exemple ci-dessus, nous avons utilisé `home.getFragment("astroItem")` pour récupérer le **fragment** `astroItem` défini dans le **template** `home.html`, puis nous avons passé l'objet `Astronaut` en utilisant la méthode `data` pour rendre ce **fragment** avec les données de l'astronaute nouvellement ajouté.

### Utilisation avec HTMX

Afin de rendre le tout vraiment dynamique, il vous reste à créer un formulaire dans votre **template** `astronaut.html` pour permettre l'ajout d'un nouvel astronaute, et à utiliser du **JavaScript** pour envoyer une requête **Ajax** à votre endpoint `add` lorsque le formulaire est soumis, afin de mettre à jour la liste des astronautes sans recharger la page entière. Plus simplement, vous pouvez utiliser du **HTMX** pour gérer cette interaction de manière plus fluide et sans avoir à écrire du code **JavaScript** personnalisé.

Par exemple :

```html title="astronaut.html"
<ul id="astroList">
    {#for astro in astronauts}
        {#fragment id=astroItem}
            <li>{astro.name}</li>
        {/fragment}
    {/for}
</ul>

<form hx-post="/add" hx-target="#astroList" hx-swap="beforeend">
    <input type="text" name="name" placeholder="Nom de l'astronaute" required />
    <button type="submit">Ajouter</button>
</form>
```

Dans ce formulaire, nous utilisons les attributs `hx-post`, `hx-target` et `hx-swap` de **HTMX** pour envoyer une requête **POST** à l'endpoint `/add` lorsque le formulaire est soumis, et pour insérer le nouveau **fragment** rendu à la fin (`beforeend`) de la liste des astronautes (`#astroList`) sans recharger la page entière.

## Template ou Fragment ?

Pour distinguer clairement quand utiliser un **template** ou un **fragment**, il est important de comprendre que les **templates** définissent généralement une vue ou un composant autonome (page, layout, ou partial réutilisable).

Les **fragments**, eux, représentent une <Underline>portion d'un template existant</Underline> que l'on souhaite pouvoir rendre séparément. Ils sont utiles lorsqu'une partie d'une page doit parfois être renvoyée seule (par exemple pour une mise à jour partielle avec HTMX) sans créer un nouveau fichier de template.

Si une section de votre interface est <Underline>réutilisée dans plusieurs pages</Underline> ou possède une <Underline>identité claire de composant</Underline>, il est généralement préférable de la définir dans un **template séparé** (souvent appelé partial). Cela améliore la lisibilité et la réutilisabilité.

En revanche, si vous avez une <Underline>sous-partie d'une page spécifique</Underline> qui doit parfois être rendue indépendamment du reste du template, il peut être plus approprié de la définir comme un **fragment** à l'intérieur du template existant.

Voici un tableau récapitulatif pour vous aider à choisir entre un **template** et un **fragment** :


| Situation | Template séparé | Fragment dans le template | Pourquoi |
|---|---|---|---|
| Bloc réutilisé dans plusieurs pages | ✅ Oui | ❌ Non | Un composant partagé doit avoir son propre fichier |
| Bloc spécifique à une seule page | ❌ Pas nécessaire | ✅ Oui | Pas besoin de créer un fichier juste pour ça |
| Bloc rendu seul depuis Java (endpoint dédié) | ✅ Très bien | ✅ Possible | Les deux fonctionnent |
| Gros bloc HTML complexe | ✅ Oui | ❌ Éviter | Lisibilité et maintenance |
| Mise à jour partielle HTMX | ✅ Très bien | ✅ Très bien | Les deux marchent |
| Bloc utilisé uniquement dans une page mais rerendu souvent | ⚖️ Au choix | ⚖️ Au choix | Question de lisibilité |
| Besoin de séparer clairement la structure de page | ✅ Oui | ❌ Non | Meilleure organisation |


<Aside type="tip">
    Un fragment n’est pas un composant réutilisable par défaut : c’est avant tout un point de découpage dans un template existant.
</Aside>
